### 3. Three "Winning" Visualizations (Copy-Paste Ready)
Judges love these three specific chart types because they show "Analytic Depth." Have these snippets ready to adapt.

#### A. The "Correlation" Chart (Dual Axis)
**Use when:** They ask "Does Weather affect Sales?" or "Do Discounts drive Volume?"
*Why it wins:* It proves a relationship between two different metrics.

```python
# In app/dashboard.py

fig = make_subplots(specs=[[{"secondary_y": True}]])

# Bar: The Core Metric (e.g., Sales)
fig.add_trace(
    go.Bar(x=df['date'], y=df['sales'], name="Sales", marker_color=COLORS['primary'], opacity=0.6),
    secondary_y=False
)

# Line: The External Factor (e.g., Temperature/Inflation)
fig.add_trace(
    go.Scatter(x=df['date'], y=df['temperature'], name="Temp", line=dict(color=COLORS['secondary'], width=3)),
    secondary_y=True
)

fig.update_layout(title="Impact of Temperature on Sales")
st.plotly_chart(fig, use_container_width=True)
```

Great framing for a data engineer discussion.

For your current project size and context, Airflow + Spark would likely introduce more cost/complexity than value.

**Possible Issues If You Use Airflow + Spark Now**
1. Over-engineering risk  
You have one pipeline with local files/API and manageable volume. Airflow + Spark is heavier than needed.

2. Infra setup overhead  
You need scheduler, metadata DB, executor setup, cluster manager, monitoring, retries, secrets handling.

3. Slower development loop  
Local DuckDB scripts are fast to iterate. Spark job + orchestration debugging is slower during hackathon/demo work.

4. Operational burden  
Airflow DAG failures, dependency drift, Spark version compatibility, cluster tuning all add maintenance load.

5. Cost and resource usage  
Spark cluster (even small) is expensive compared with local DuckDB for this data shape.

6. Complexity in portability  
Reviewers can run your current project quickly. Requiring Airflow/Spark reduces reproducibility for evaluators.

**Why DuckDB Is a Good Choice Here**
1. Warehouse-like SQL, local file  
You get analytical SQL performance without running a server.

2. Excellent for medium-scale BI  
Your tables (millions of rows) fit DuckDB very well for aggregations and dashboard queries.

3. Very low friction  
No cluster, no scheduler service required to prove pipeline design and data modeling quality.

4. Fast prototyping + production-minded structure  
You still implemented Bronze/Silver/Gold, quality checks, and fail-fast rules, which are production concepts.

5. Reproducible demos  
Anyone can run `python bi-solution-hack.` and `streamlit run ...` in one machine and see the same result.

**When Airflow + Spark Would Make Sense**
1. Many independent pipelines with scheduling dependencies.
2. Very large distributed workloads (data no longer fits single-machine processing efficiently).
3. Multi-team platform requirements (central orchestration, lineage, SLA management).
4. Streaming + near real-time heavy compute.

**Good way to say it in your demo**
“I intentionally chose DuckDB for local analytical speed and reproducibility. The architecture is already medallion-based, so if scale or scheduling complexity grows, the same model can be migrated to Airflow orchestration and Spark/distributed compute later.”
